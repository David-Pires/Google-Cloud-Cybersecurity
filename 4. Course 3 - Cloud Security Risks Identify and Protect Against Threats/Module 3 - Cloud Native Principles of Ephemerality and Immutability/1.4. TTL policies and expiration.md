

# ⏳ Time-To-Live (TTL) Policies | Políticas de Tempo de Vida

### 🇺🇸 English

A **TTL policy** is a security and management setting that determines how long a data asset remains accessible. It is a numerical value (usually a timestamp) that acts as an expiration timer. Once the TTL expires, the cloud provider automatically makes the asset inaccessible or deletes it, ensuring that sensitive data isn't exposed longer than necessary.

### 🇧🇷 Português

Uma **política de TTL** é uma configuração de segurança e gerenciamento que determina por quanto tempo um ativo de dados permanece acessível. É um valor numérico (geralmente um registro de data/hora) que funciona como um cronômetro de expiração. Assim que o TTL expira, o provedor de nuvem torna o ativo inacessível ou o exclui automaticamente, garantindo que dados sensíveis não fiquem expostos por mais tempo do que o necessário.

---

## 🛠️ Operational Workflow | Fluxo Operacional (Firestore Example)

### 1. Creating a Policy | Criando uma Política

* **🇺🇸 English:** You define a "Collection Group" and a specific field that contains the timestamp. The cloud service monitors this field and triggers the expiration.
* **🇧🇷 Português:** Você define um "Grupo de Coleção" e um campo específico que contém o registro de data/hora. O serviço de nuvem monitora esse campo e aciona a expiração.

### 2. Stopping/Deleting a Policy | Parando/Excluindo uma Política

* **🇺🇸 English:** If you need the data for longer than planned, you can delete the TTL policy. This doesn't delete the data; it simply removes the "expiration timer," making the data stay available indefinitely until manually removed.
* **🇧🇷 Português:** Se você precisar dos dados por mais tempo do que o planejado, pode excluir a política de TTL. Isso não exclui os dados; apenas remove o "cronômetro de expiração", fazendo com que os dados permaneçam disponíveis indefinidamente até serem removidos manualmente.

## 🛡️ Analyst's Insight | Insight do Analista

**🇺🇸 English:**
From a security perspective, TTL is a powerful tool for **Data Minimization**. If an attacker gains unauthorized access to your database, they can only steal what is currently there. By using TTL for session tokens, temporary passwords, or short-term logs, you ensure that "old" data is automatically purged, leaving nothing for an attacker to find in the future.

**🇧🇷 Português:**
*Sob a perspectiva de segurança, o TTL é uma ferramenta poderosa para a **Minimização de Dados**. Se um invasor obtiver acesso não autorizado ao seu banco de dados, ele só poderá roubar o que estiver lá no momento. Ao usar TTL para tokens de sessão, senhas temporárias ou logs de curto prazo, você garante que dados "antigos" sejam expurgados automaticamente, não deixando nada para um invasor encontrar no futuro.*

---

### 🚀 Key Takeaways | Pontos Chave

* **Inaccessibility:** After expiration, assets are no longer reachable by users or entities. / *Após a expiração, os ativos não são mais alcançáveis por usuários ou entidades.*
* **Flexibility:** Policies can be stopped at any time by deleting the TTL rule. / *As políticas podem ser interrompidas a qualquer momento excluindo a regra de TTL.*
* **Service Variation:** TTL behaves differently depending on the service (Firestore vs. DynamoDB vs. Cloud Storage). / *O TTL se comporta de forma diferente dependendo do serviço.*
