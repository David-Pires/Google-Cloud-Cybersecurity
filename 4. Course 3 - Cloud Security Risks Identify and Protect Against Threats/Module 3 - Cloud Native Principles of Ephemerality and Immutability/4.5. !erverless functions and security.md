

# ⚡ Serverless Functions & Security | Funções Serverless e Segurança

### 🇺🇸 English

A **Serverless Function** (also known as Function-as-a-Service or FaaS) is a bundle of code that runs in a fully managed environment. Unlike containers, which require scaling via orchestration (Kubernetes), serverless functions scale **automatically and dynamically** based on demand. You don't manage servers; you only pay for the execution time of your code.

### 🇧🇷 Português

*Uma **Função Serverless** (também conhecida como Function-as-a-Service ou FaaS) é um pacote de código que roda em um ambiente totalmente gerenciado. Ao contrário dos containers, que exigem escalonamento via orquestração (Kubernetes), as funções serverless escalam **automática e dinamicamente** com base na demanda. Você não gerencia servidores; você paga apenas pelo tempo de execução do seu código.*

---

## ⚖️ Containers vs. Serverless

| Feature / Recurso | **Containers (Docker/K8s)** | **Serverless (Cloud Functions/Lambda)** |
| --- | --- | --- |
| **Management / Gestão** | You manage the OS/Orchestration. / Você gere o SO/Orquestração. | Provider manages everything. / O provedor gere tudo. |
| **Scaling / Escala** | Manual or auto-scaling rules. / Regras de auto-scaling. | Automatic & Instant. / Automática e Instantânea. |
| **Lifecycle / Vida** | Long-running (Stateful/Stateless). / Longa duração. | Short-lived (Ephemeral). / Curta duração (Efêmera). |
| **Pricing / Preço** | Pay for provisioned resources. / Paga por recursos reservados. | Pay per execution/request. / Paga por execução/requisição. |

---

## 🛡️ The Shared Responsibility Model | Modelo de Responsabilidade Compartilhada

No modelo Serverless, a linha de demarcação é muito clara:

* **Cloud Provider (Google/AWS/Azure):** 🇺🇸 *Responsible for hardware, virtualization layer, and the runtime environment.* / 🇧🇷 Responsável pelo hardware, camada de virtualização e ambiente de runtime.
* **Cloud User (You):** 🇺🇸 *Responsible for the **code**, its security, authentication, and the privileges (IAM) assigned to it.* / 🇧🇷 Responsável pelo **código**, sua segurança, autenticação e os privilégios (IAM) atribuídos a ele.

---

## 🔒 Serverless Security Best Practices | Melhores Práticas

Apesar de não haver servidor para "invadir" via SSH, existem riscos críticos:

1. **Least Privilege (IAM):**
* 🇺🇸 *Do NOT share service accounts between functions. Each function should have its own identity with the absolute minimum permissions needed.*
* 🇧🇷 NÃO compartilhe contas de serviço entre funções. Cada função deve ter sua própria identidade com o mínimo absoluto de permissões necessárias.


2. **Network Security (Ingress/Egress):**
* 🇺🇸 *Control who can trigger the function (Ingress) and what external APIs the function can talk to (Egress).*
* 🇧🇷 Controle quem pode acionar a função (Entrada) e com quais APIs externas a função pode falar (Saída).


3. **Code Security:**
* 🇺🇸 *Since the code is the only thing you control, it must be scanned for vulnerabilities and hardcoded secrets.*
* 🇧🇷 Como o código é a única coisa que você controla, ele deve ser escaneado em busca de vulnerabilidades e segredos (senhas) expostos.

## 🛡️ Analyst's Insight | Insight do Analista

**🇺🇸 English:**
The biggest security "trap" in serverless is **Over-privileged Service Accounts**. Because it's easier to give a function "Admin" rights to make it work quickly, many developers ignore the principle of least privilege. As a security professional, you must audit **IAM Roles** to ensure a function that only needs to *write* to a database doesn't have the permission to *delete* the entire database.

**🇧🇷 Português:**
*A maior "armadilha" de segurança em serverless são as **Contas de Serviço com excesso de privilégios**. Como é mais fácil dar direitos de "Admin" a uma função para fazê-la funcionar rápido, muitos desenvolvedores ignoram o princípio do menor privilégio. Como profissional de segurança, você deve auditar as **IAM Roles** para garantir que uma função que só precisa *escrever* em um banco de dados não tenha permissão para *deletar* o banco inteiro.*

---

### 🚀 Key Takeaway

**"Serverless doesn't mean 'no servers'; it means 'someone else's servers'. You still own the security of your code."**
*(Serverless não significa 'sem servidores'; significa 'servidores de outra pessoa'. Você ainda é o dono da segurança do seu código.)*
