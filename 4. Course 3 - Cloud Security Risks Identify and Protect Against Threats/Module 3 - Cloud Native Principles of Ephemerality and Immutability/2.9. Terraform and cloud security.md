

# 🤖 Terraform & Security Automation | Terraform e Automação de Segurança

### 🇺🇸 English

Terraform allows security professionals to build a **Security Baseline** directly into the code. By defining infrastructure as immutable and version-controlled, you reduce the risk of "backdoors" and ensure that access controls are never left wide open. The goal is to avoid **Configuration Drift**, where manual changes make the environment less secure over time.

### 🇧🇷 Português

*O Terraform permite que profissionais de segurança construam uma **Linha de Base de Segurança (Security Baseline)** diretamente no código. Ao definir a infraestrutura como imutável e controlada por versão, você reduz o risco de "backdoors" e garante que os controles de acesso nunca fiquem escancarados. O objetivo é evitar o **Desvio de Configuração (Configuration Drift)**, onde alterações manuais tornam o ambiente menos seguro com o tempo.*

---

## 🔍 Continuous Auditing Checklist | Checklist de Auditoria Contínua

Ao usar Terraform, você deve auditar constantemente estes pontos críticos:

* **RBAC (Role-Based Access Control):** Quem tem permissão para quê?
* **Secrets Management:** Existem senhas ou chaves de API "hardcoded" (escritas diretamente) nos scripts?
* **Network Traffic:** As regras de **Ingress** (entrada) e **Egress** (saída) estão seguindo o princípio do menor privilégio?
* **Privileged Access:** Quem possui acesso Root ou permissões administrativas?

## 🛠️ Terraform Concepts for Security | Conceitos Vitais do Terraform

Para gerenciar a segurança em escala, o Terraform utiliza estruturas organizacionais:

| Concept / Conceito | 🇺🇸 Security Function | 🇧🇷 Função de Segurança |
| --- | --- | --- |
| **Workspaces** | Acts as a security boundary for variables, logs, and state. | Age como uma fronteira de segurança para variáveis, logs e estado. |
| **Projects** | Groups related workspaces to manage permissions at scale. | Agrupa workspaces relacionados para gerenciar permissões em escala. |
| **Permissions** | Granular control (Read, Plan, Write, Admin) for teams. | Controle granular (Leitura, Plano, Escrita, Admin) para equipes. |

## 🛡️ Best Practices | Melhores Práticas

### 🇺🇸 English

1. **Avoid Manual Changes:** If you change something via the Cloud GUI (Console), the next time the Terraform script runs, it will **revert** your manual change. Always update the script first.
2. **Use Provider Modules:** Use resources curated by the Cloud Provider (Google, AWS), as they are usually pre-secured.
3. **Test and Commit:** Always test your scripts and keep them in **Version Control** (like Git) for traceability and accountability.

### 🇧🇷 Português

1. **Evite Mudanças Manuais:** Se você alterar algo via GUI (Console), na próxima vez que o script Terraform rodar, ele **reverterá** sua mudança manual. Sempre atualize o script primeiro.
2. **Use Módulos do Provedor:** Utilize recursos curados pelo provedor de nuvem (Google, AWS), pois eles geralmente já vêm pré-configurados com segurança.
3. **Teste e Comite:** Sempre teste seus scripts e mantenha-os em **Controle de Versão** (como Git) para rastreabilidade e responsabilidade.

## 🛡️ Analyst's Insight | Insight do Analista

**🇺🇸 English:**
Terraform is your "Source of Truth." If a security breach happens, the first place you should look is your version control history. If the infrastructure code hasn't changed, but the environment has, you've detected a **Configuration Drift**—which could be a sign of a manual error or a malicious actor at work.

**🇧🇷 Português:**
*O Terraform é sua "Fonte da Verdade". Se ocorrer uma violação de segurança, o primeiro lugar onde você deve olhar é o histórico do seu controle de versão. Se o código da infraestrutura não mudou, mas o ambiente mudou, você detectou um **Desvio de Configuração** — o que pode ser sinal de um erro manual ou de um ator malicioso em ação.*
