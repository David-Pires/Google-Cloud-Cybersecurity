Focado em **GCP**, configurar o **Cloud Load Balancing** via **Terraform** para **Failover Global** é uma das estratégias mais robustas de **Business Continuity**. Isso garante que, se uma região inteira do Google (ex: `us-east1`) cair, o tráfego seja redirecionado automaticamente para outra (ex: `southamerica-east1`), mantendo o **RTO** próximo de zero.


### 🌐 Estratégia de Failover com Cloud Load Balancing (HTTPS)

O segredo do failover na GCP está no **Global Load Balancer**, que utiliza um único IP externo para distribuir tráfego entre múltiplos **Backend Services** em regiões diferentes.

#### 1. Conceitos Chave para o Terraform:

* **Health Checks:** O balanceador monitora a "saúde" das instâncias. Se o backend da Região A parar de responder, ele redireciona o tráfego para a Região B.
* **Backend Service:** Onde definimos os grupos de instâncias das diferentes regiões.
* **URL Map:** O roteiro que define para onde o tráfego deve ir.

---

### 💻 Exemplo de Código Terraform (Failover Multi-Regional)

```hcl
# 1. Health Check: O "Termômetro" do sistema
resource "google_compute_health_check" "http_health_check" {
  name               = "check-app-status"
  check_interval_sec = 5
  timeout_sec        = 5
  http_health_check {
    port = 80
  }
}

# 2. Backend Service Global com Failover
resource "google_compute_backend_service" "failover_backend" {
  name                  = "web-backend-global"
  protocol              = "HTTP"
  load_balancing_scheme = "EXTERNAL"
  health_checks         = [google_compute_health_check.http_health_check.id]

  # Backend Principal (Ex: Brasil)
  backend {
    group          = var.instance_group_region_a
    balancing_mode = "UTILIZATION"
    capacity_scaler = 1.0
  }

  # Backend de Disaster Recovery (Ex: EUA)
  backend {
    group          = var.instance_group_region_b
    balancing_mode = "UTILIZATION"
    capacity_scaler = 1.0 
  }
  # O GCP distribuirá para o mais próximo, mas se um falhar, 
  # o tráfego migra automaticamente para o outro.
}

# 3. URL Map: O Roteador Global
resource "google_compute_url_map" "url_map" {
  name            = "global-lb-url-map"
  default_service = google_compute_backend_service.failover_backend.id
}

```

---

### 🛡️ Benefícios para o Plano de Recuperação (DR)

1. **Failover Automatizado**: O Terraform provisiona a infraestrutura que detecta a falha e desvia o tráfego sem intervenção humana.
2. **Continuidade de Negócio**: Os usuários finais nem percebem que houve um desastre na região principal, preservando a **Confiança do Usuário**.
3. **Simulação de Desastre**: Você pode testar seu plano de BCDR simplesmente diminuindo o `capacity_scaler` de uma região para 0 no Terraform e observando o tráfego migrar.

---

### 📈 Nota Consultiva para seu Repositório:

> "O uso de **Load Balancing Failover** reduz drasticamente o **RTO (Recovery Time Objective)**, pois a recuperação do tráfego é quase instantânea. Contudo, é vital garantir que o **RPO (Recovery Point Objective)** esteja alinhado, assegurando que o banco de dados na Região B esteja sincronizado com a Região A via **Replication**".
